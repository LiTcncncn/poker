# 玩法组合系统设计方案

## 一、系统概述

### 1.1 设计目标
设计一个灵活、可扩展的玩法组合系统，允许通过不同维度的组合创造出丰富的游戏玩法，提升游戏的可玩性和多样性。

### 1.2 核心思想
将玩法拆解为多个可组合的维度，每个维度有多个选项。**重要：玩法配置由策划人员预先配置，所有维度都是确定的，不使用随机策略。** 这样可以确保玩法平衡和可控，达到设计的确定性。

### 1.3 超级牌解锁系统
- **52张超级牌对应52种玩法**：每张超级牌都有对应的解锁玩法
- **所有配置都是确定的**：牌池、Joker几率、任务、额外规则都由策划人员预先配置
- **玩法列表存储**：开发时，玩法列表单独作为JSON文件存储，以便后续编辑

---

## 二、玩法组合维度

### 2.1 张数维度（Card Count）
**选项：** 3、4、5、6

**说明：**
- 决定每局游戏使用的牌数
- 影响牌型判断的复杂度
- 不同张数对应不同的牌型规则

**实现考虑：**
- 3张：只能形成高牌、一对、三条、顺子、同花（**不包括同花顺和皇家同花顺**）
- 4张：可以形成高牌、一对、两对、三条、四条、顺子、同花（**不包括同花顺和皇家同花顺**）
- 5张：标准德州扑克牌型（高牌、一对、两对、三条、顺子、同花、葫芦、四条、同花顺、皇家同花顺）
- 6张：扩展玩法，可能形成更复杂的组合

**重要规则：**
- 3张和4张玩法中，**不存在同花顺牌型**（包括皇家同花顺）
- 4张玩法中，**可以形成四条牌型**

---

### 2.2 牌池维度（Card Pool）
**选项：**
- **基础牌池**：白色牌 + 超级牌（不包括品质牌池：绿色、蓝色、紫色）
- **全部牌池**：包括所有品质的牌（白色、绿色、蓝色、紫色、超级牌）

**说明：**
- **牌池由策划人员预先配置，不使用随机策略**
- 基础牌池：适合新手或简单玩法，只使用基础牌和超级牌
- 全部牌池：包含品质牌，增加策略性
- **当前52个超级牌玩法全部使用"全部牌池"**

**实现考虑：**
- 需要区分"品质牌池"的概念
- 品质牌池包括：绿色、蓝色、紫色品质的牌
- 基础牌池 = 白色牌 + 超级牌
- 全部牌池 = 基础牌池 + 品质牌池

---

### 2.3 Joker 几率维度（Joker Probability）
**选项：**
- **0%**：不出现 Joker 牌
- **3%**：Joker 出现概率 3%
- **8%**：Joker 出现概率 8%（joker++模式）
- **10%**：Joker 出现概率 10%

**说明：**
- **Joker几率由策划人员预先配置，不使用随机策略**
- 所有52个超级牌玩法都有确定的Joker几率配置
- 支持多种Joker概率值（0%、3%、8%、10%等）

**实现考虑：**
- 需要支持动态设置 Joker 概率（根据配置值）
- Joker概率在玩法配置中明确指定

---

### 2.4 额外规则维度（Extra Rules）
**选项（可多选或组合）：**

#### 2.4.1 成牌规则
- **3张成顺/成花**：3张牌即可形成顺子或同花（不包括同花顺和皇家同花顺）
- **4张成顺/成花**：4张牌即可形成顺子或同花（不包括同花顺和皇家同花顺）
- **5张成顺/成花**：标准规则，需要5张牌

#### 2.4.2 花色限制
- **红色同花**：只有红心（♥）和方块（♦）可以形成同花
- **黑色同花**：只有黑桃（♠）和梅花（♣）可以形成同花
- **无限制**：所有花色都可以形成同花

#### 2.4.3 数值限制规则
- **单数成顺**：只有单数牌可以形成顺子（A、3、5、7、9、J、K）
- **双数成顺**：只有双数牌可以形成顺子（2、4、6、8、10、Q）

#### 2.4.4 赔率加成规则
- **牌型赔率+1**：指定牌型的赔率增加1倍（如两对赔率+1、三条赔率+1、顺子赔率+1、同花赔率+1、葫芦赔率+1等）
- 可以针对不同牌型设置不同的赔率加成

#### 2.4.5 其他规则
- **允许同花顺**：在"n张成顺/成花"规则下，是否允许同花顺和皇家同花顺
- **特殊牌型**：可以添加其他特殊规则（待扩展）

**实现考虑：**
- 需要修改牌型判断逻辑，支持"n张成顺/成花"
- 需要支持花色限制规则
- 规则之间可能有冲突，需要定义优先级

---

### 2.5 任务需求维度（Task/Objective）
**选项：**
- **高牌 n 手**：需要完成 n 次高牌
- **一对 n 手**：需要完成 n 次一对
- **两对 n 手**：需要完成 n 次两对
- **三条 n 手**：需要完成 n 次三条
- **顺子 n 手**：需要完成 n 次顺子
- **单数顺子 n 手**：需要完成 n 次单数顺子（只有单数牌形成的顺子）
- **双数顺子 n 手**：需要完成 n 次双数顺子（只有双数牌形成的顺子）
- **同花 n 手**：需要完成 n 次同花
- **葫芦 n 手**：需要完成 n 次葫芦
- **四条 n 手**：需要完成 n 次四条
- **5条 n 手**：需要完成 n 次5条
- **同花顺 n 手**：需要完成 n 次同花顺
- **皇同 n 手**：需要完成 n 次皇家同花顺
- **其他任务**：可以扩展更多任务类型

**说明：**
- 任务定义了玩家的目标
- **任务难度由策划人员配置**：不同超级牌的任务次数可能不同（如高牌3手、一对3/5/10手、两对3/5/8/20手等）
- **无任务奖励**：任务是解锁下一个超级牌的条件，不提供奖励
- **所有任务类型、目标次数都由策划人员预先配置**

**实现考虑：**
- 需要任务追踪系统
- 需要任务完成判定逻辑
- 任务配置存储在配置文件中，由策划人员维护
- 任务进度用于超级牌解锁，不提供奖励

---

## 四、系统设计

### 4.1 玩法配置数据结构

**重要：开发时，玩法列表单独作为JSON文件存储，以便后续编辑。**

```typescript
interface GameModeConfig {
  id: string;                    // 玩法唯一标识
  name: string;                  // 玩法名称
  superCardId: string;           // 对应的超级牌ID（如 'spades_A'）
  cardCount: 3 | 4 | 5 | 6;      // 张数（由策划配置）
  cardPool: 'basic' | 'all';     // 牌池类型（全部玩法使用'all'）
  jokerProbability: number;      // Joker概率（0、0.03、0.08、0.10等，由策划配置）
  extraRules: ExtraRule[];       // 额外规则列表（由策划配置）
  task: TaskConfig;              // 任务配置（由策划配置）
  enabled: boolean;              // 是否启用
}

interface ExtraRule {
  type: 'n_card_straight' | 'n_card_flush' | 'color_flush' | 'allow_straight_flush' | 
        'odd_straight' | 'even_straight' | 'multiplier_bonus';
  value?: number;                // n的值（如3、4），或赔率加成倍数
  color?: 'red' | 'black';       // 花色限制
  handType?: 'two_pairs' | 'three_of_a_kind' | 'straight' | 'flush' | 'full_house';  // 赔率加成的牌型
}

interface TaskConfig {
  type: 'high_card' | 'one_pair' | 'two_pairs' | 'straight' | 'odd_straight' | 'even_straight' | 'flush' | 
        'three_of_a_kind' | 'full_house' | 'four_of_a_kind' | 'five_of_a_kind' |
        'straight_flush' | 'royal_flush';
  target: number;                // 目标次数 n（由策划人员配置，不同超级牌可能不同）
  // 注意：无任务奖励，任务是解锁超级牌的条件
}
```

### 4.2 玩法配置原则

**重要：玩法配置由策划人员预先配置，所有维度都是确定的，不使用随机策略。**

#### 4.2.1 配置原则
1. **所有玩法配置由策划人员预先定义**，存储在JSON文件中（`gameModes.json`）
2. **不允许玩家自定义玩法**，只能选择策划配置好的玩法
3. **规则、任务都由策划人员配置**，确保玩法平衡和可控
4. **所有配置都是确定的**：
   - 牌池：由策划人员配置（52个玩法全部使用"全部牌池"）
   - Joker概率：由策划人员配置（支持0%、3%、8%、10%等）
   - 不使用随机策略，确保设计的确定性
5. 配置时需要遵循兼容性规则，避免不合理的组合

#### 4.2.2 兼容性规则
1. **张数与成牌规则兼容性：**
   - 3张玩法：只能使用"3张成顺/成花"规则，**不存在同花顺牌型**
   - 4张玩法：可以使用"3张成顺/成花"或"4张成顺/成花"规则，**不存在同花顺牌型**，但**可以形成四条**
   - 5张玩法：可以使用"3张成顺/成花"、"4张成顺/成花"或标准规则，包括同花顺
   - 6张玩法：可以使用所有成牌规则

2. **任务与张数兼容性：**
   - 3张玩法：只能完成高牌、一对、三条、顺子、同花任务（**不包括同花顺和皇家同花顺**）
   - 4张玩法：可以完成高牌、一对、两对、三条、四条、顺子、同花任务（**不包括同花顺和皇家同花顺**）
   - 5张玩法：可以完成所有标准牌型任务（包括同花顺和皇家同花顺）
   - 6张玩法：可以完成所有牌型任务（可能包括扩展牌型）

3. **花色限制与任务兼容性：**
   - 红色/黑色同花限制：只能用于"同花"相关任务
   - 其他任务不受花色限制影响

### 6.4 超级牌解锁机制

#### 6.4.1 解锁逻辑变更
**原逻辑：** 在基础玩法下，用$购买解锁超级牌

**新逻辑：** 解锁每个超级牌，都是一个在玩法池中的玩法，通过规则+任务+$达到解锁条件，解锁新超级牌，之后获得新的超级牌目标，即新玩法+任务+$目标。

#### 6.4.2 解锁流程
1. **超级牌对应玩法**：每个超级牌对应一个特定的玩法配置
2. **解锁顺序**：根据现有逻辑从♠️A到♦️K依次解锁，必须完成前一张超级牌的解锁条件才能解锁下一张
3. **解锁条件**：玩家需要在该玩法中完成：
   - **规则要求**：满足玩法配置的规则要求
   - **任务要求**：完成指定的任务（任务难度由策划人员配置）
   - **$要求**：一次性支付一定数量的$（钻石/货币）
4. **解锁条件显示**：
   - 超级牌待解锁时，主界面的超级牌区改为解锁条件区
   - 分别显示各项解锁条件的进度（规则、任务、$）
   - 全部解锁条件完成可解锁本阶段超级牌
5. **解锁后**：解锁新超级牌后，在主界面显示下一个超级牌目标（按顺序：♠️A→♠️2→...→♦️K）

#### 6.4.3 数据结构设计
```typescript
interface SuperCardUnlockConfig {
  superCardId: string;           // 超级牌ID（如 'spades_A'），按顺序从♠️A到♦️K
  gameModeId: string;            // 对应的玩法ID
  unlockOrder: number;            // 解锁顺序（1-52，对应♠️A到♦️K）
  unlockConditions: {
    task: TaskConfig;            // 任务要求（任务难度由策划人员配置）
    cost: number;                // 解锁所需的$数量（一次性支付）
    // 规则要求由玩法配置中的 extraRules 定义
  };
  nextSuperCardId?: string;      // 下一个要解锁的超级牌ID（按顺序）
}

interface UnlockProgress {
  superCardId: string;           // 超级牌ID
  ruleCompleted: boolean;        // 规则要求是否完成
  taskProgress: {
    current: number;              // 当前任务进度
    target: number;               // 目标任务次数
    completed: boolean;            // 任务是否完成
  };
  costPaid: boolean;              // $是否已支付
  unlocked: boolean;              // 是否已解锁
}
```

- **任务进度**：用 "当前进度/目标次数" 格式显示（如 "3/5" 表示已完成3次，目标5次）
- **$和解锁按钮**：显示所需$数量和解锁按钮（和现有一致）

**显示示例：**

## 九、总结

本方案提供了一个灵活、可扩展的玩法组合系统设计，通过多个维度的组合可以创造出丰富的游戏玩法。

### 9.1 核心要点
1. **52张超级牌对应52种玩法**，所有配置都是确定的，不使用随机策略
2. **玩法配置由策划人员预先配置**，存储在JSON文件中（`gameModes.json`），以便后续编辑
3. **不允许玩家自定义玩法**，只能选择策划配置好的玩法
4. **所有维度都是确定的**：
   - 牌池：全部使用"全部牌池"
   - Joker概率：由策划人员配置（支持0%、3%、8%、10%等）
   - 不使用随机策略，确保设计的确定性
5. **3、4张玩法不存在同花顺牌型**，4张玩法支持四条牌型
6. **任务类型包括**：高牌、一对、两对、三条、顺子、同花、葫芦、四条、5条、同花顺、皇同、单数顺子、双数顺子等
7. **任务难度由策划人员配置**：不同超级牌的任务次数可能不同，无任务奖励
8. **超级牌解锁机制**：
   - 每个超级牌对应一个玩法，通过规则+任务+$达到解锁条件
   - **解锁顺序**：从♠️A到♦️K依次解锁，必须完成前一张才能解锁下一张
   - 超级牌待解锁时，主界面的超级牌区改为解锁条件区
   - 主界面显示：目标超级牌、玩法名、任务进度（3/5格式）、$和解锁按钮
   - 分别显示各项解锁条件的进度，全部完成可解锁
   - $消耗为一次性支付
   - 解锁后在主界面显示下一个超级牌目标（按顺序）
9. **成就系统限制**：
   - 成就系统的顺子、同花、同花顺、皇家同花顺都需要5张牌
   - 超级三张（3张）和超级四张（4张）玩法无法完成以上成就
   - 只有5张玩法才能完成这些成就
