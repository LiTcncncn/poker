# Roguelike 界面改版方案（本关目标 / 进度条 / 补牌与手数 / 结算区）

## 背景与约束

- **仅手机竖屏适配**：主内容窄栏单列（现状 `max-w-[390px]`），宽屏也保持同一列版心。
- **高度占位不增加**：本关目标区 + 结算区这几个区域的“垂直占位”要与现状一致（整体页面高度体验不变），优先通过**信息密度/排版重组**实现。
- **交互新增**：原“公式数值来源”先隐藏；点击新的「计算公式」区域弹出弹窗显示数值来源。
- **公式展示形态调整**：公式改为两个矩形容器，中间是“✖️”：
  - 左矩形：`$（本手牌$计数）`
  - 右矩形：`x（本手牌+倍计数）`
  - 整个区域代表「计算公式」入口（可点击）

## 现状代码定位（已确认）

- **本关目标 / 进度条 / 补牌 / 剩余手数**：`roguelike/src/components/StageView.tsx` 中 `L264-L331` 一段。
- **结算区（牌型 + 总收益 + 公式 + 技能条目 chips）**：`roguelike/src/components/HandView.tsx` 的 `SettlementBlock`（`L16-L122`）。
- 目前公式展示为：
  - \((cardScoreSum + skillAddedScore) × (baseMultiplier + skillAddedMultiplier) × independentMultiplier = finalGold\)
  - “数值来源”并未单独展开，只有 `skillLog` 的 chips（最多 6 个）。

## 改版目标（UI/信息结构）

### 顶部「本关目标」区（不加高）

目标：在不增加高度的前提下，增强可读性与“当前处境”辨识，信息更“可扫读”。

现状结构（3 行）：
1) 目标文案 + 精英词缀 chip
2) 进度条
3) 下方一行：补牌（含点点） + 剩余手数

改版建议（仍保持 3 行，不新增行高）：
- **第 1 行**（信息更聚合）：
  - 左：`本关目标 $当前/$目标`（数字用 tabular-nums，突出 $当前）
  - 右：精英/Boss 词缀 chip（保持可点击展开，不改交互）
- **第 2 行**：进度条保持现状（包含预览层与 glow 逻辑，不动视觉高度）
- **第 3 行**（更“状态化”的短标签）：
  - 左：`补牌 2/3` + 点点（点点保留，但缩短文案）
  - 右：`剩余手 4`（数字加粗）

高度控制手段：
- 仍用现有 `gap-1.5` / `text-xs` / `h-2` 级别，不新增额外 padding。
- 仅微调文案长度与对齐方式（例如去掉括号内冗余文字）。

### 结算区（不加高，但信息更“分区”）

目标：结算区保持现有垂直节奏（牌型与 +$ 大字、下面公式、再下面 skillLog chips），但把“公式”改为可点击的矩形块组合，并把“数值来源”移到弹窗里。

#### 结算区结构（建议保持 3 段，不新增段落）

1) **牌型 + 总收益**（保持现状字号与占位）
   - 仍是：`手牌名` + `+$finalGold`

2) **计算公式入口（新视觉）**（替换现状一行公式文本）
   - 布局：水平居中，允许换行（现状 `flex-wrap`），但尽量在一行内完成。
   - 形态：
     - 左矩形（Score Box）：`$（本手牌$计数）`
     - 中间：`✖️`
     - 右矩形（Mult Box）：`x（本手牌+倍计数）`
   - 点击反馈：整块可点击（`button` 或 `role="button"`），按压态/聚焦态轻微高亮，不改变高度。
   - 解释文案：矩形内部用两行排版（上大下小）会增加高度；因此建议**单行内容**：
     - 左矩形文本：`$ 123`（副标题用更小字号放在同一行尾部：`（本手牌）`）
     - 右矩形文本：`x 45`（尾部：`（本手牌+倍）`）
   - 若你坚持“`$（本手牌$计数）`”与“`x（本手牌+倍计数）`”必须完整显示：
     - 也可做成：矩形里上方显示 `$ 123` / `x 45`，下方显示括号说明，但需要通过 **更紧的 line-height（leading-4）** 与 **更小字号（text-[10px]）** 把总高度压回现状（需要你确认是否接受更小字）。

3) **技能来源 chips**（保留现状，但定位为“本手触发摘要”）
   - 保持 `max-h-[72px]` 与展示 6 个 + 余数不变。
   - “公式数值来源”不再在此处展示（避免加高）。

## “数值来源”弹窗（新）

### 触发

- 点击结算区的「计算公式」块时打开弹窗（预览态 `preview=true` 不可点击，保持现状 `pointer-events-none`）。

### 弹窗承载内容（建议 2 层信息密度）

**上半：汇总（对齐玩家心智）**
- 标题：`数值来源`
- 第一行：`总收益 = $左值 ✖️ x右值`（与新公式一致）

**下半：可折叠明细（避免过长）**
- `$（本手牌$计数）` 明细（可折叠）：
  - **牌面分合计（计分牌）**：来自 `baseCardSum`
  - **牌型基础$**：来自 `handTypeBaseScore`
  - **技能+$**：逐条列出（来自 skillLog 中 `addedScore` 项）
  - 若存在“剩余手奖励并入技能$”（`applySavedHandsStageWinBonus`）：单独一条说明其来源（需要引擎侧提供标记）
- `x（本手牌+倍计数）` 明细（可折叠）：
  - **牌型基础倍**：来自 `stats.multiplier`
  - **技能+倍**：逐条列出（skillLog 中 `addedMultiplier` 项）
  - **独立乘区**：逐条列出（skillLog 中 `multiplyFactor` 项；如果多次叠乘，需要显示乘法链或最终因子）

> 备注：当前 `HandResult` 只有 `cardScoreSum`、`skillAddedScore`、`multiplierTotal`、`skillAddedMultiplier`、`independentMultiplier`、`skillLog`，不足以把 `$` 与 `x` 的来源拆得“完全可解释”。因此弹窗要落地，需要补充“结算 breakdown”数据（见下一节）。

### 弹窗样式（手机习惯，不影响主界面高度）

- 形式：底部弹层（与现有精英词缀弹层一致的结构：`fixed inset-0 ... items-end ... max-w-[390px] rounded-t-2xl`）。
- 高度：`max-h-[min(70vh,520px)]` + 内部滚动。
- 关闭：点击遮罩关闭 + 底部一个“关闭”按钮（与现有一致）。

## 数据与实现建议（为 UI 服务的最小改动）

### 1) 给结算结果补充 breakdown（推荐）

新增一个结构（命名仅建议）：

- `HandResult.formulaBreakdown?: { ... }`
  - `baseCardSum`：计分牌牌面$合计（目前在 `skillEngine.ts` 内部变量 `baseCardSum`，未输出）
  - `handTypeBaseScore`：牌型基础$（目前是 `stats.baseScore`，未输出）
  - `handTypeBaseMultiplier`：牌型基础倍（`stats.multiplier`，可由 `multiplierTotal - skillAddedMultiplier` 推得，但直接输出更稳）
  - `skillAddsScore: Array<{ name: string; value: number; sourceId: string }>`
  - `skillAddsMultiplier: Array<{ name: string; value: number; sourceId: string }>`
  - `independentFactors: Array<{ name: string; factor: number; sourceId: string }>`
  - `savedHandsBonus?: { savedHands: number; addedScore: number; bySkills: ... }`（若你希望说明“通关剩余手奖励”的并入逻辑）

实现落点：
- `evaluateHandWithSkills()` 内已经生成 `skillLog`；只需在返回前把 `baseCardSum`、`stats.baseScore`、`stats.multiplier`、以及把 `skillLog` 分组映射成三类数组即可。
- `_commitScore()` 将 `evalResult` 转成 `handResult` 时把 breakdown 带到 `HandResult`（类型需要扩展）。
- Hold 预览 `previewHandResult` 同样会拿到 breakdown（但 UI 预览不弹窗即可）。

### 2) 结算区 UI 改造点（仅布局与交互）

文件建议：
- `roguelike/src/components/HandView.tsx`
  - 替换现有 `L40-L74` 的公式行，改成「可点击公式块」组件（例如 `FormulaSummaryButton`）。
  - 新增弹窗组件（例如 `FormulaSourceModal`），接收 `HandResult`（或其 breakdown）渲染。

交互状态：
- `preview === true`：公式块不可点击（保持现状 preview 整体 `pointer-events-none` 即可）。
- `preview === false`：公式块点击打开弹窗。

### 3) 顶部目标区：不加高的微调点

文件：
- `roguelike/src/components/StageView.tsx`
  - 保留 3 行结构与现有 `gap`/`text` 尺寸级别。
  - 改文案与对齐（例如“补牌（a/b）”改为“补牌 a/b”），点点保留但不扩高度。

## 我需要你确认/回答的问题（决定能否“落地且不加高”）

1) **新公式矩形容器的文字必须按你写的完整形式显示吗？**
   - 选项 A：矩形内只显示精简版（`$ 123（本手牌）` / `x 45（本手牌+倍）`），保证高度不变最稳。
   - 选项 B：严格显示“`$（本手牌$计数）`”与“`x（本手牌+倍计数）`”这两段完整文案（更长），可能需要更小字号或允许换行，存在“变高”的风险。

2) **“本手牌$计数”与“本手牌+倍计数”两侧的数值，你希望展示哪两个数？**
   - 我按现有公式理解：
     - 左值应是 \(cardScoreSum + skillAddedScore\)（或更细：`baseCardSum + handTypeBaseScore + skillAddedScore`）
     - 右值应是 \(multiplierTotal\)（并且独立乘区 `independentMultiplier` 要不要并入右值？）
   - 你现在的新表达“右边容器是 x（本手牌+倍计数）”看起来更像“总倍率”，但现有还有 `independentMultiplier` 这一段：请确认它归入哪里。

3) **独立乘区（`independentMultiplier`）在新 UI 的归属**
   - 选项 A：并入右矩形（右值直接显示 `multiplierTotal * independentMultiplier`，弹窗里再拆开）
   - 选项 B：仍单独显示第三段（但这会破坏你要求“两个矩形容器”的视觉）
   - 选项 C：右矩形显示“基础倍 + 技能倍”，弹窗里额外显示“独立乘区”，总收益计算仍按三段（UI 只展示两段）

4) **数值来源弹窗的详尽程度**
   - 选项 A（轻量）：只展示分组后的 skillLog（技能+$ / 技能+倍 / 独立乘区），以及基础项 2～3 行（牌面合计、牌型基础$、牌型基础倍）。
   - 选项 B（重）：需要把每个 skillLog 条目精确映射到“触发条件/参与牌/来源规则”，这会要求引擎侧产出更结构化的日志（开发量明显更大）。

5) **通关剩余手奖励并入技能$（`applySavedHandsStageWinBonus`）是否必须在来源里单列说明？**
   - 若必须：我们需要在结果里携带 `savedHands` 与该 bonus 的明细，否则 UI 只能“猜”。
