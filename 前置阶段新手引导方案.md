# 前置阶段新手引导方案

## 一、概述

前置阶段是游戏的新手引导系统，通过逐步解锁牌位（从1张到5张），让玩家熟悉基础牌型和游戏机制，为正式游戏做准备。

---

## 二、核心机制

### 2.1 解锁流程

| 阶段 | 当前牌数 | 解锁费用 | 解锁后牌数 |
|------|---------|---------|-----------|
| 初始 | 1张 | - | - |
| 第1次解锁 | 1张 | $100 | 2张 |
| 第2次解锁 | 2张 | $200 | 3张 |
| 第3次解锁 | 3张 | $300 | 4张 |
| 第4次解锁 | 4张 | $400 | 5张 |

### 2.2 解锁规则

- **解锁条件**：仅需金币足够即可解锁
- **解锁方式**：用户手动点击"牌+1"按钮
- **解锁提示**：达到解锁金额时，按钮显示呼吸光效果，引导用户点击
- **解锁效果**：解锁后立即显示新牌位（扣住状态）

### 2.3 前置阶段完成标志

- **完成条件**：第一次翻5张牌时，前置阶段自动完成
- **完成效果**：
  - "牌+1"按钮变为"自动"按钮
  - 抽卡按钮等正式功能解锁显示
  - 进入正式游戏流程

---

## 三、翻牌规则适配

### 3.1 各牌数下的可用牌型

| 牌数 | 可用牌型 | 说明 |
|------|---------|------|
| 1张 | 高牌 | 仅显示简单结算，无动画 |
| 2张 | 高牌、一对 | - |
| 3张 | 高牌、一对、三条 | - |
| 4张 | 高牌、一对、两对、三条 | **不包含四条** |
| 5张 | 全部牌型 | 正式游戏规则 |

### 3.2 结算显示规则

- **1张牌**：显示简单结算板，无动画效果
- **2-5张牌**：按正常规则显示结算板和动画（对子及以上）

---

## 四、特殊机制：固定触发队列

### 4.1 设计目标

通过固定触发机制，让玩家在2、3、4张牌时，能够体验到对子、两对、三条等牌型，作为新手的刺激体验。

### 4.2 触发队列设计

#### 2张牌阶段（解锁后第1-3次翻牌）
- 第1次：高牌
- 第2次：**一对**（刺激点）
- 第3次：高牌

#### 3张牌阶段（解锁后第1-3次翻牌）
- 第1次：高牌
- 第2次：一对
- 第3次：**三条**（刺激点）

#### 4张牌阶段（解锁后第1-4次翻牌）
- 第1次：高牌
- 第2次：一对
- 第3次：**两对**（刺激点）
- 第4次：**三条**（刺激点）

### 4.3 实现方式

- 使用固定队列，按顺序触发
- 每个牌数阶段独立计数
- 队列结束后，恢复随机生成
- 确保刺激但不突兀，符合新手体验节奏

## 五、UI/UX 设计

### 5.1 牌位显示

- **显示规则**：根据已解锁牌数显示对应数量的牌位，其他隐藏
  - 1张牌：只显示1个牌位，其他4个隐藏
  - 2张牌：显示2个牌位，其他3个隐藏
  - 以此类推

### 5.2 按钮状态

#### "牌+1"按钮
- **显示文字**：显示所需金币，如"牌+1 ($100)"
- **呼吸光效果**：金币足够时，按钮显示呼吸光动画，引导用户点击
- **禁用状态**：金币不足时，按钮禁用（灰色）

#### "自动"按钮
- **显示时机**：前置阶段完成后显示
- **功能**：正常的自动翻牌功能

### 5.3 功能按钮显示

- **抽卡按钮**：前置阶段隐藏，完成后显示
- **其他功能按钮**：按正常规则显示

## 六、数据持久化

### 6.1 状态保存

- **保存内容**：已解锁牌数（`unlockedCardSlots: 1-5`）
- **保存时机**：每次解锁后立即保存
- **恢复机制**：用户刷新页面后，自动恢复前置阶段进度

### 6.2 前置阶段完成标志

- **状态字段**：`tutorialStage: 'pre' | 'complete'`
- **完成触发**：第一次翻5张牌时，自动设置为 `'complete'`
- **完成效果**：不再显示"牌+1"按钮，解锁所有正式功能

## 七、技术实现要点

### 7.1 状态管理

```typescript
interface GameState {
  // 前置阶段相关
  tutorialStage: 'pre' | 'complete'; // 前置阶段 | 完成
  unlockedCardSlots: number; // 已解锁牌位数量 (1-5)
  tutorialFlipCount: Record<number, number>; // 各牌数阶段的翻牌计数
}
```

### 7.2 核心函数修改

#### `flipCards()` 函数
- 根据 `unlockedCardSlots` 决定翻牌数量
- 根据 `tutorialFlipCount` 应用固定触发队列
- 1-4张时只判定基础牌型（跳过顺子、同花等）

#### `evaluateResolvedHand()` 函数
- 支持动态牌数（1-5张）
- 1-4张时跳过需要5张的牌型判定
- 4张时不判定四条

#### 新增 `unlockCardSlot()` 函数
- 检查金币是否足够
- 扣除解锁费用
- 增加 `unlockedCardSlots`
- 重置该牌数阶段的翻牌计数

### 7.3 UI 组件修改

#### `App.tsx`
- 根据 `tutorialStage` 显示/隐藏抽卡按钮
- 根据 `tutorialStage` 和 `unlockedCardSlots` 显示"牌+1"或"自动"按钮
- 根据 `unlockedCardSlots` 显示对应数量的牌位
- 实现"牌+1"按钮的呼吸光效果

#### `Card.tsx`
- 支持动态牌数显示（隐藏未解锁牌位）

## 九、实现细节

### 9.1 固定触发队列实现

`generateTutorialHand` 函数根据牌数和翻牌次数生成特定牌型：

- **2张牌**：第2次翻牌生成一对
- **3张牌**：第2次翻牌生成一对，第3次翻牌生成三条
- **4张牌**：第2次翻牌生成一对，第3次翻牌生成两对，第4次翻牌生成三条

如果生成失败（牌堆中缺少所需牌），则回退到随机生成。

### 9.2 牌型判定限制

在 `evaluateResolvedHand` 函数中：
- 1-4张牌时，跳过需要5张的牌型（顺子、同花、同花顺、皇家同花顺、葫芦）
- 4张牌时，不判定四条（`canHaveFourOfAKind = cardCount === 5`）

### 9.3 UI 显示逻辑

- **牌位显示**：根据 `unlockedCardSlots` 显示对应数量的牌位
- **按钮显示**：
  - 前置阶段：显示"牌+1"按钮（金币足够时显示呼吸光效果）
  - 完成后：显示"自动"按钮和抽卡按钮
- **动画显示**：1张牌时的高牌不显示动画，只显示简单结算板

## 九、注意事项

1. **数据迁移**：老用户首次加载时，自动设置为 `tutorialStage: 'complete'`，`unlockedCardSlots: 5`
2. **结算动画**：1张牌时只显示简单结算，无动画效果
3. **牌型判定**：4张牌时不判定四条，避免过于复杂
4. **固定队列**：确保队列刺激但不突兀，符合新手体验节奏
5. **按钮引导**：呼吸光效果要明显但不刺眼，有效引导用户点击
